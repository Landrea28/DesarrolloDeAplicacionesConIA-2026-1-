### 1. El riesgo de "romper" en las pruebas

Cuando haces una prueba de penetración (hacking ético), tu trabajo es atacar. Y como en cualquier ataque, hay daños colaterales. No es que seas descuidado, es que así funciona el proceso.

- **El problema de los datos falsos:** Al probar formularios, llenas la base de datos con "basura" (nombres falsos, correos inexistentes).
    
- **El peligro de los ataques reales:** Si lanzas un ataque de inyección (meter código malicioso), podrías corromper la base de datos de verdad.
    
- **Consecuencia fatal:** En un entorno real (producción), esto es un desastre. En un entorno de laboratorio, si no puedes "resetearlo", perderás horas o días reconstruyéndolo todo.
    

### 2. Tu "Botón de Pánico": La Recuperación

Para no perder el tiempo ni el juicio, necesitas una red de seguridad. El texto nos da dos opciones principales:

- **Entornos Virtuales (El ideal):** Usar máquinas virtuales te permite sacar **"Snapshots"** (fotos del estado actual). Si rompes algo, haces clic y regresas al pasado en segundos.
    
- **Copias de Respaldo (El plan B):** Si el sistema es físico y no se puede virtualizar, necesitas un respaldo completo. Es más lento, pero te salva la vida.

## 3.🔑 Palabras Clave

- **Entorno de Producción:** El sistema real que usan los clientes o empleados (¡Prohibido romper aquí!).
    
- **Entorno de Laboratorio:** Un lugar seguro y controlado para hacer pruebas sin riesgos externos.
    
- **Virtualización:** Tecnología que permite crear una computadora "ficticia" dentro de otra.
    
- **Snapshot (Instantánea):** Una captura que guarda el estado exacto de un sistema en un momento dado para poder volver a él si algo falla.
